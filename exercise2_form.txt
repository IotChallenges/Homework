Logic
We consider the following three scenarios:
In-Mouth: While the denture is worn, we want to put the device to deep sleep and not
transmit any beacon signals.
Out of Mouth, Stationary: When the denture is resting on a nightstand or in a glass,
only a few initial beacons are necessary to estimate the location.
Out of Mouth, Moving: If the denture is not worn but still moving, the beacons need
to be emitted continuously and often.
Sensing
To detect our scenarios of interest, we need two additional sensors.
• Temperature Sensor: Used to detect if the denture is inside the mouth by measuring
the body’s heat
• Acceleration Sensor: Used to detect motion if the denture is outside the mouth.
Pseudocode
resting_transmits = 0

setup() {
  init_temp_sensor()
  init_accelerometer(Wake_on_Motion_Interrupt) // send interrupt if motion is detected
}

loop() {
  temperature = read_temperature()
  motion_detected = read_last_motion_state()

  if temperature > 34.0:
    resting_transmits = 0
    disable_ble_beacons()
    // acts as return statement, no further code is executed
    sleep(5 minutes)

  if motion_detected:
    resting_transmits = 0
    enable_ble_beacons()
    set_ble_beacon_frequency(1 Hz)
    sleep(1 second)

  // we have transmitted our beacon 5 times and will not emit any further beacons until movement is detected
  if resting_transmits > 4:
    disable_ble_beacons()
    resting_transmits = 0
    // no need to wake up before motion is detected, only 1 day sleep as a fail-safe
    sleep(1 day)
  else:
    enable_ble_beacons()
    // transmit beacon every minute for five minutes
    set_ble_beacon_frequency(1/60 Hz)
    resting_transmits = resting_transmits + 1
    sleep(1 minute)
}
Hardware
The hardware for these dentures must fit inside a dental prosthesis and consume very
little power. Thus, the following components were selected.
- MCU: Nordic Semiconductor nRF52810
- Accelerometer: STMicroelectronics LIS2DW12
Energy Savings
We assume, that using the RF module and thus the beacon transmission draw the majority
of power and that the MCU’s and sensor’s power draw is negligible.
Original implementation:
1 d = 24 · 60 · 60 s = 86 400 s
86 400 s · 0.1 Hz = 8640 beacons
For our implementation, we assume the following usage characteristics over a single
day:
• In-Mouth: 16 h
• Outside Mouth Moving: 10 min
• Outside Mouth Resting: 7 h50 min
Improved implementation:
n_mouth = 0 beacons
n_resting = 5 beacons
n_moving = 10 min · 1 Hz = 600 beacons
n_total = nmouth + nresting + nmoving = 605 beacons
This results in the following energy savings:
605 ÷ 8640 ≈ 0.07 = 7%
Despite our aggressive beacon transmission during movement, we transmit 7% of the
original beacon count and thus only use 7% of the original power draw.
